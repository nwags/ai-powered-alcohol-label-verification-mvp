# Data Models

This document is authoritative for API request/response shapes.

Code must not introduce additional field names or status values without updating this document.

This document defines the canonical request and response shapes for the prototype.

Its purpose is to keep the API, service layer, UI layer, tests, and Codex-generated code aligned.

If implementation details differ from this file, this file should be treated as the source of truth unless another requirements document explicitly overrides it.

---

# Design Principles

1. Keep schemas small and explicit.
2. Prefer required fields only where truly necessary.
3. Return deterministic status values.
4. Preserve raw OCR text for transparency.
5. Use `review` instead of inventing certainty.

---

# Field Enum Types

These enums define the supported verification fields and statuses.

## VerificationField

Allowed verification field names:

- `brand_name`
- `class_type`
- `alcohol_content`
- `net_contents`
- `bottler_producer`
- `country_of_origin`
- `government_warning`

These names should be used consistently in:

- request payloads
- response payloads
- domain models
- tests
- UI rendering

---

## FieldStatus

Allowed field status values:

- `match`
- `normalized_match`
- `mismatch`
- `review`

Definitions:

- `match`  
  Exact match between application data and parsed label value.

- `normalized_match`  
  Equivalent after normalization such as case folding, whitespace cleanup,
  punctuation normalization, unit normalization, or proof-to-ABV conversion.

- `mismatch`  
  Clear difference between application data and parsed label value.

- `review`  
  Insufficient certainty to determine correctness automatically.

No other field status values are allowed.

---

## OverallStatus

Allowed overall result values:

- `match`
- `normalized_match`
- `mismatch`
- `review`

Recommended aggregation logic:

- If any required field is `review`, overall status should be `review`.
- Else if any required field is `mismatch`, overall status should be `mismatch`.
- Else if any field is `normalized_match`, overall status may be `normalized_match`.
- Else overall status is `match`.

This aggregation may be refined later, but all code should begin with this rule.

---

## LabelType

Allowed label-type hint values:

- `unknown`
- `brand_label`
- `other_label`

Definitions:

- `unknown`  
  Default. Keep full-field aggregation behavior.
- `brand_label`  
  Prioritize `brand_name`, `class_type`, and `alcohol_content` for overall status.
- `other_label`  
  Prioritize `government_warning`, `bottler_producer`, `net_contents`, and `country_of_origin` for overall status.

In MVP, `label_type` is a routing hint for aggregation only. All field results remain visible.

# Request Schemas

The prototype supports two primary request types:

1. single-label analysis
2. batch analysis

---

## Single Label Analysis Request

Endpoint:

`POST /api/v1/analyze`

Content type:

`multipart/form-data`

Parts:

### File Part

- `image`  
  Required. Label image file.

Supported formats should include at minimum:

- `.png`
- `.jpg`
- `.jpeg`
- `.webp` (optional if easy to support)

### Form Field Part

- `application_json`  
  Required. JSON string containing the submitted application fields.
- `label_type`  
  Optional. Defaults to `unknown`.

Example:

```json
{
  "brand_name": "Stone's Throw",
  "class_type": "Whiskey",
  "alcohol_content": "45% Alc./Vol.",
  "net_contents": "750 mL",
  "bottler_producer": "Bottled by Example Spirits Co., Bethlehem, PA",
  "country_of_origin": "United States",
  "government_warning": "GOVERNMENT WARNING: (1) According to the Surgeon General, women should not drink alcoholic beverages during pregnancy because of the risk of birth defects. (2) Consumption of alcoholic beverages impairs your ability to drive a car or operate machinery, and may cause health problems."
}
```

# Canonical Internal Single Analysis Request Model

This is the internal Python/domain shape the application should normalize to after input parsing.

```json
{
  "request_id": "optional-string-or-generated-uuid",
  "label_type": "unknown",
  "application": {
    "brand_name": "string or null",
    "class_type": "string or null",
    "alcohol_content": "string or null",
    "net_contents": "string or null",
    "bottler_producer": "string or null",
    "country_of_origin": "string or null",
    "government_warning": "string or null"
  },
  "options": {
    "enable_preprocessing": true,
    "enable_visualization": true
  }
}
```

## Notes

- `request_id` may be omitted by the client and generated by the server.
- `label_type` defaults to `unknown` when omitted or invalid.
- `country_of_origin` may be null for non-import products.
- `government_warning` may be omitted by the client only if the system uses a built-in canonical warning string for comparison. If that is done, document it clearly in code and tests.
- `options` may be added later; keep defaults simple.

# Batch Analysis Request

Endpoint:

`POST /api/v1/batch/analyze`

Recommended input shape for MVP:

`multipart/form-data`

Parts:

- `batch_file`
   Required. CSV or JSON describing application records.
- `images_archive`
   Optional but recommended. ZIP of label images keyed by filename.
- `label_type`
   Optional. Defaults to `unknown` and applies as a batch-level hint.
- Multiple uploaded images may be used instead of that is simpler for MVP.

## Recommended CSV Columns

- `record_id`
- `image_filename`
- `brand_name`
- `class_type`
- `alcohol_content`
- `net_contents`
- `bottler_producer`
- `country_of_origin`
- `government_warning`

## Recommended JSON Array Shape

```json
[
  {
    "record_id": "001",
    "image_filename": "label_001.jpg",
    "brand_name": "Stone's Throw",
    "class_type": "Whiskey",
    "alcohol_content": "45% Alc./Vol.",
    "net_contents": "750 mL",
    "bottler_producer": "Bottled by Example Spirits Co., Bethlehem, PA",
    "country_of_origin": "United States",
    "government_warning": "GOVERNMENT WARNING: ..."
  }
]
```

Batch mode is optional for MVP, but if implemented it should reuse the single-label analysis pipeline internally.

## Response Field Definitions

`request_id`

Server-generated or client-provided identifier for traceability.

`overall_status`

One of:

- `match`
- `normalized_match`
- `mismatch`
- `review`

`timing_ms`
End-to-end processing time in milliseconds.

`ocr.lines`
Ordered OCR segments with confidence and bounding boxes.

`parsed`
Structured values extracted from OCR text.

This should contain best-effort parsing output even when matching is uncertain.

`field_results`
The main reviewer-facing comparison results.

Each field result should include:

- `status`
- `submitted_value`
- `detected_value`
- `notes`

`artifacts`
Optional generated assets such as annotated images.

`errors`
Non-fatal processing issues. This should be an array of strings or structured error objects.

For MVP, strings are acceptable.

# Minimal Field Result Model

Each verification field in `field_results` should follow this shape:

```json
{
  "status": "review",
  "submitted_value": "string or null",
  "detected_value": "string or null",
  "notes": "string or null"
}
```

This must remain consistent across all fields.

---

# Minimal Parsed Field Models

The parsed object may vary slightly by field type, but should remain stable.

## ParsedTextValue

Used for simple text fields such as:

- `brand_name`
- `class_type`
- `bottler_producer`
- `country_of_origin`

Shape:

```json
{
  "value": "string or null"
}
```

## ParsedAlcoholContent

Shape:

```json
{
  "raw": "string or null",
  "abv_percent": 45.0,
  "proof": 90.0
}
```

Rules:

- `raw` may be null if not detected
- `abv_percent` may be null
- `proof` may be null

## ParsedNetContents

Shape:

```json
{
  "raw": "string or null",
  "milliliters": 750
}
```

Rules:

- `raw` may be null
- `milliliters` may be null

## ParsedWarning

Shape:

```json
{
  "value": "string or null",
  "detected": true
}
```

Rules:

- `value` may be partial if OCR is incomplete
- `detected` indicates whether warning-like text was found

# Error Response Schema

When the request is invalid or cannot be processed, return a structured error response.

Recommended shape:

```json
{
  "error": {
    "code": "invalid_image",
    "message": "Uploaded file is not a supported image format."
  }
}
```

Recommended error codes:

- `invalid_request`
- `invalid_image`
- `image_too_large`
- `ocr_failed`
- `internal_error`

Do not expose stack traces in API responses.

---

# Batch Analysis Response

If batch mode is implemented, recommended response shape:

```json
{
  "batch_id": "string",
  "summary": {
    "total": 10,
    "match": 3,
    "normalized_match": 2,
    "mismatch": 1,
    "review": 4
  },
  "results": [
    {
      "record_id": "001",
      "request_id": "req-001",
      "overall_status": "review"
    }
  ],
  "errors": []
}
```

Batch mode should avoid inventing a separate result model unless clearly needed.

Prefer reusing the single-label response shape per record.

# UI Rendering Guidance

The UI should render `field_results` as the canonical reviewer-facing table.

Suggested columns:

- Field
- Submitted Value
- Detected Value
- Status
- Notes

This ensures backend and frontend stay aligned.

---

# Testing Guidance

Tests should validate at minimum:

1. only allowed enum values are returned
2. `field_results` contains only canonical field names
3. OCR text is present in successful responses
4. parsing models allow null values safely
5. invalid images return structured errors
6. aggregation to `overall_status` is deterministic
